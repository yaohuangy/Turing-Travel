# **途灵**旅行（**Turing Travel**）需求文档

## 1. 项目概述

一个 AI 驱动的智能旅行规划系统。用户输入目的地、出发/结束日期、预算、人数、兴趣偏好，系统自动生成一份结构化的多日旅行行程，包含每日景点安排、地图路线、天气信息、预算明细，并支持行程保存、单日编辑、Markdown/PDF 导出。

## 2. 技术栈

| 层级 | 技术 |
|---|---|
| 后端框架 | Python FastAPI + Uvicorn |
| 数据校验 | Pydantic v2 |
| 数据库 | SQLite（开发）/ PostgreSQL（生产），ORM 用 SQLAlchemy 2.0 |
| LLM 编排 | LangChain，模型用阿里 DashScope 的 qwen-max |
| 向量数据库 | ChromaDB，Embedding 模型用 DashScope text-embedding-v4 |
| Rerank | DashScope qwen3-rerank（Cross-encoder） |
| 缓存 | Redis（可选，需支持优雅降级） |
| 外部 API | 高德地图 Web 服务（地理编码、POI 搜索、路线规划、天气） |
| PDF 导出 | ReportLab |
| Markdown 导出 | Jinja2 模板 |
| 测试 | pytest |
| 前端框架 | Vue 3 + TypeScript + Vite |
| UI 组件库 | Ant Design Vue 4 |
| 前端地图 | 高德地图 JS API |
| HTTP 客户端 | Axios |

## 3. 功能需求

### 3.1 行程生成（核心功能）

**输入表单：**
- 目的地：文本输入，支持中英文城市名
- 出发日期 + 结束日期：日期选择器，自动计算天数
- 预算档位：经济型 / 舒适型 / 豪华型 三档
- 出行人数：数字输入
- 兴趣偏好：多选标签（自然风光、历史文化、美食探店、亲子乐园、休闲度假、户外运动）
- 额外需求：自由文本输入框，可选

**生成逻辑：**
- 将表单参数组装为 prompt，调用 qwen-max 生成结构化的多日行程 JSON
- 生成结果须包含：每日景点列表（名称、建议游玩时间、简要描述）、每日餐饮推荐、住宿建议
- 每次生成需包含 RAG 检索增强：从本地攻略知识库中检索与目的地相关的旅行攻略片段，注入 LLM prompt 作为参考上下文
- RAG 检索需经过 Query Rewrite（LLM 改写查询）和 Cross-encoder Rerank（精排）两步优化
- 生成行程后，自动调用高德 API 为每个景点补充：经纬度坐标、详细地址
- 自动计算相邻景点间的驾车距离和耗时
- 自动查询目的地未来天气预报，匹配到每一天
- 根据预算档位和景点数量自动估算费用明细（交通、住宿、餐饮、门票、其他）

**输出结构（JSON Schema）：**
```
{
  destination: string,
  start_date: string,
  end_date: string,
  days: [{
    day_index: number,
    date: string,
    weather: { condition: string, temp_high: number, temp_low: number } | null,
    spots: [{
      name: string,
      address: string,
      location: { lng: number, lat: number },
      visit_duration: string,
      description: string,
      image_url: string | null,
      poi_id: string | null
    }],
    meals: [{ type: "breakfast"|"lunch"|"dinner", name: string, description: string }],
    hotel: { name: string, location: string, estimated_price: number } | null
  }],
  budget: {
    total: number,
    transportation: number,
    accommodation: number,
    meals: number,
    tickets: number,
    other: number
  }
}
```

### 3.2 行程保存与历史管理

- 用户可保存当前生成的行程，保存时需提供行程名称（自动填充目的地+日期）
- 保存后存入 SQLite trip_records 表
- 历史页面展示所有已保存行程的列表（卡片形式），每条显示：名称、目的地、日期范围、总预算、保存时间
- 支持点击进入查看详情（加载到 Result 页面）
- 支持删除行程

### 3.3 单日行程智能编辑

- 在 Result 页面，用户可选中某一天，用自然语言描述修改需求
- 例如："把第二天的午餐换成火锅"、"下午加一个博物馆"、"删除第三个景点"
- 后端接收编辑指令后，将该天的现有行程 + 用户指令一起发给 qwen-max，只重写该天的内容，不动其他天
- 编辑后自动重新调用高德 API 为新景点补充坐标和地址

### 3.4 天气查询

- 独立的天气查询接口，输入城市名，返回未来天气预报
- 天气数据优先从 Redis 缓存读取（TTL 2 小时），缓存未命中则调高德天气 API
- 天气信息包含：日期、天气状况、最高温、最低温、风向风力

### 3.5 文档导出

**Markdown 导出：**
- 用 Jinja2 模板将行程 JSON 渲染为 Markdown 文件
- 包含：标题、每日行程（景点列表 + 天气 + 餐食 + 住宿）、预算明细表
- 返回 Markdown 文本，前端触发下载

**PDF 导出：**
- 用 ReportLab 生成 PDF 文件
- 包含与 Markdown 相同的内容，但使用正式排版（中文字体、标题层级、表格）
- 返回 PDF 文件流，前端触发下载

### 3.6 地图可视化（前端）

- 使用高德地图 JS API 在页面上渲染交互式地图
- 将行程中所有景点的坐标标为 Marker 点，按天数用不同颜色区分
- 每天的景点之间绘制驾车路线折线
- 点击 Marker 弹出信息窗，显示景点名称、地址、图片（如有）、描述
- 地图默认视野自动适配所有景点的包围盒

## 4. 后端 API 设计

### 4.1 行程接口

**POST /api/trip/generate**
- 功能：生成新行程
- 请求体：
  ```json
  {
    "destination": "大理",
    "start_date": "2026-06-01",
    "end_date": "2026-06-03",
    "budget_level": "comfort",
    "travelers": 2,
    "preferences": ["自然风光", "美食探店"],
    "extra_requirements": "希望多安排一些洱海边的景点"
  }
  ```
- 响应：完整的行程 JSON（结构见 3.1 输出结构）

**POST /api/trip/edit**
- 功能：编辑单日行程
- 请求体：
  ```json
  {
    "trip_id": "xxx",
    "day_index": 2,
    "edit_instruction": "把午餐换成过桥米线，下午加一个博物馆"
  }
  ```
- 响应：修改后的完整行程 JSON

**POST /api/trip/save**
- 功能：保存行程
- 请求体：
  ```json
  {
    "name": "大理三日游",
    "destination": "大理",
    "start_date": "2026-06-01",
    "end_date": "2026-06-03",
    "itinerary": { ... }
  }
  ```
- 响应：`{ "trip_id": "uuid" }`

**GET /api/trips**
- 功能：获取已保存行程列表
- 响应：`[{ trip_id, name, destination, start_date, end_date, total_budget, saved_at }, ...]`

**GET /api/trip/{trip_id}**
- 功能：获取单个行程详情
- 响应：完整的行程 JSON

**DELETE /api/trip/{trip_id}**
- 功能：删除行程
- 响应：`{ "deleted": true }`

### 4.2 导出接口

**GET /api/export/{trip_id}/markdown**
- 功能：导出 Markdown 文本
- 响应：Markdown 文本（Content-Type: text/markdown）

**GET /api/export/{trip_id}/pdf**
- 功能：导出 PDF 文件
- 响应：PDF 文件流（Content-Type: application/pdf）

### 4.3 天气接口

**GET /api/weather/forecast?city=大理**
- 功能：查询天气预报
- 响应：
  ```json
  [{
    "date": "2026-06-01",
    "condition": "晴",
    "temp_high": 26,
    "temp_low": 16,
    "wind": "西南风 3级"
  }, ...]
  ```

## 6. 前端页面与组件

### 6.1 页面结构

**App.vue**：顶部标签导航（"规划行程"、"行程结果"、"历史记录"），三个标签分别对应三个页面。

**Home.vue（规划行程页）**：
- 表单布局，居中卡片式
- 目的地输入框（AutoComplete，建议城市名）
- 日期范围选择器（RangePicker）
- 预算档位单选框
- 人数数字输入
- 偏好标签多选
- 额外需求文本域
- "生成行程"按钮，点击后调 POST /api/trip/generate，loading 状态，完成后跳转到 Result 页面并携带行程数据

**Result.vue（行程结果页）**：
- 左侧面板：按天折叠面板（Collapse），每展开一天显示该天的景点列表、餐食、住宿、天气卡片
- 右上区域：地图组件（AmapTripMap），展示所有景点标记和路线
- 右下区域：预算明细表格和饼图
- 底部操作栏：保存按钮、Markdown 导出按钮、PDF 导出按钮
- 编辑入口：每天的折叠面板内有一个"编辑"按钮，点击弹出 Modal，输入自然语言修改指令，提交调 POST /api/trip/edit

**History.vue（历史记录页）**：
- 行程卡片列表（Card Grid），每条显示名称、目的地、日期、预算
- 卡片点击进入该行程的 Result 页面
- 卡片右上角删除按钮，确认后调 DELETE /api/trip/{id} 并刷新列表

### 6.2 核心组件

**AmapTripMap.vue**：
- Props：接收 DayPlan 数组
- 初始化高德地图实例
- 遍历所有 DayPlan，按天为每个 Spot 添加 Marker（不同天数不同颜色图标）
- 每天的 Spot 之间绘制 Polyline（驾车路线）
- 点击 Marker 弹出 InfoWindow，显示景点名称、地址、描述、图片
- 设置地图视野 fitView 包含所有 Marker

## 7. RAG 检索管道

### 7.1 知识库

- 本地存储多个热门目的地的 Markdown 格式旅行攻略文档（放 backend/data/ 目录）
- 每个文档包含：景点介绍、美食推荐、住宿建议、交通提示、行程参考
- 目标覆盖 10-15 个中国热门旅游城市

### 7.2 文档入库流程

- 读取每个 Markdown 文件，按"## "二级标题切分为 chunks
- 每个 chunk 调用 DashScope text-embedding-v4 生成 1536 维向量
- 存入 ChromaDB，collection 名为 "travel_guides"，metadata 包含：source_file, city, title, chunk_index

### 7.3 在线检索流程（retriever.py）

1. 接收原始查询 query 和目的地 city
2. Query Rewrite：
   - 主方案：调用 qwen-max，用 Few-shot prompt 将用户偏好转为更精准的检索查询，同时指定目的地过滤
   - 回退方案：如果 LLM Rewrite 失败，用规则方式提取目的地关键词 + 偏好关键词拼接
3. 向量检索：用改写后的 query 在 ChromaDB 做 top_k 检索（带 city 元数据过滤）
4. Noise 过滤：对检索结果做相关性阈值过滤，筛掉低分 chunk
5. Cross-encoder Rerank：用 qwen3-rerank 对候选 chunks 精排，返回 top_n（n 可配置）
6. 结果缓存：相同 query + city 的检索结果缓存到 Redis（TTL 1 小时）

### 7.4 评估体系

- eval/rag_eval_cases.json：包含每个目的地 3-5 个测试用例，每个用例有 query、city、期望命中的 chunk_id 列表
- evaluate_rag_retrieval.py：自动化评测脚本
- 指标：Top1 命中率、TopK 命中率、MRR、噪声率、跨目的地污染率、平均延迟
- 每次改动检索管道后跑全量评测，记录分数变化

## 8. 缓存策略

### 8.1 Redis 缓存项

| 缓存 Key 模式 | 内容 | TTL |
|---|---|---|
| weather:{city} | 城市天气预报 JSON | 2 小时 |
| geocode:{address} | 地理编码结果（经纬度） | 永久 |
| poi:{keyword}:{city} | POI 搜索结果 | 24 小时 |
| route:{origin_lng},{origin_lat}:{dest_lng},{dest_lat} | 路线规划结果 | 1 小时 |
| rag:{query_hash}:{city} | RAG 检索结果 | 1 小时 |
| rerank:{query_hash}:{candidate_hash} | Rerank 结果 | 1 小时 |

### 8.2 降级策略

- 所有 Redis 操作包裹 try-except
- Redis 不可用时直接调用原始 API，不抛异常，不影响主流程
- 日志记录 Redis 错误但不阻断请求

## 10. 开发顺序

### 阶段 1：项目骨架
1. 初始化后端项目：FastAPI 空项目 + 目录结构 + requirements.txt + .env 配置
2. 初始化前端项目：Vite + Vue 3 + TypeScript + Ant Design Vue 引入
3. 定义 Pydantic 数据模型（schemas.py：TripRequest、Itinerary、DayPlan、SpotItem 等）
4. 定义 SQLAlchemy 数据模型（db_models.py：TripRecord 表）
5. 前后端各自跑通，确认环境正常

### 阶段 2：核心生成链路
6. 实现 trip_planner_agent.py：构造 prompt → 调 qwen-max → 解析 JSON → 返回结构化行程
7. 实现 POST /api/trip/generate 路由，不含地图/天气/RAG，仅 LLM 生成
8. 实现 Home.vue 页面：表单 + 提交 + 跳转 Result 页面
9. 前端调通生成接口，能在结果页看到 JSON 展示

### 阶段 3：地图数据充实
10. 实现 cache_service.py：Redis 连接管理、JSON 读写、降级逻辑

12. 在 trip_service.py 中将 map_service 注入生成流程：行程生成后为每个景点调地理编码 + POI 补充坐标和地址，为每天的景点序列计算路线
13. 每条 map_service 调用都过 Redis 缓存

### 阶段 4：天气数据充实
14. 实现 weather_service.py：调高德天气 API 获取天气预报，过 Redis 缓存

### 阶段 5：存储与导出

19. 实现 export_service.py 的 Markdown 导出（Jinja2 模板 + 渲染）

26. 实现保存按钮、导出按钮（触发文件下载）

## 12. 非功能性要求

- CORS：后端允许前端开发服务器的跨域请求
- 错误处理：所有外部 API 调用加 try-except，LLM 调用加重试（最多 3 次），失败时返回明确的错误信息而非 500
- 响应时间：行程生成接口属于慢接口（预计 5-15 秒），前端需展示 loading 状态，不阻塞 UI
- 缓存降级：Redis 不可用时所有功能正常运行，仅在日志中记录
- 日志：关键节点打印日志（生成开始/结束、API 调用耗时、检索命中数），便于排查问题
